# Субагент: Документация (Правила 13-17, 19, 21-22)

## Правила

### Правило 19: About секция
- Добавить topics для поиска
- Добавить description (краткое однострочное описание)

### Правило 13: CI баджы в README
- Добавить баджи CI (CI passing)
- Добавить баджи анализаторов (CodeCov, Codacy)

### Правило 14: Описание проекта в README
- Написать несколько абзацев о проекте
- Описание должно быть понятно новичку

### Правило 15: Примеры использования
- Примеры использования с картинками/gif
- Информация о датасетах, конфигурации

### Правило 16: Инструкции по сборке
- Описать зависимости (версии SDK)
- Описать как собрать проект локально

### Правило 17: Инструкции для контрибьюторов
- Как писать баги
- Как связаться с разработчиками
- Как контрибьютить
- Где найти техническую документацию

### Правило 21: Техническая документация
- Архитектурная документация (wiki или README раздел)
- UML диаграммы при возможности

### Правило 22: Комментарии в коде
- DocString, Doxygen, Javadoc, XML Documentation
- Описания классов, функций, модулей
- Желательно: автоматическая генерация документации (Sphinx)

## Анализ

1. Проверить README.md на наличие баджей
2. Проверить описание проекта
3. Проверить примеры использования
4. Проверить инструкции по сборке
5. Проверить CONTRIBTING.md
6. Проверить docstring-и/комментарии в коде

## Результат

В результате надо вернуть табличку, каждая строчка которой соответствует одному правилу.
В табличке три столбца: номер правила, результат, описание причины.
Возможные результаты:
- `PASS` — документация полная
- `FAIL` — нарушение с описанием
- `NA` — не применимо
